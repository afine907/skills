# 用户故事编写指南

## 什么是用户故事

用户故事是从最终用户角度对功能的简短描述。

## 用户故事模板

```
作为 <用户角色>，
我希望 <功能/行为>，
以便 <价值/收益>。
```

## 示例

**反例（不好）：**
> 作为用户，我可以上传文件。

**正例（好）：**
> 作为注册用户，我希望上传头像图片，以便其他用户能够认识我。

## INVEST 原则

优秀的用户故事应符合 INVEST 原则：

| 字母 | 含义 | 说明 |
|------|------|------|
| **I** | Independent（独立） | 故事之间尽量不相互依赖 |
| **N** | Negotiable（可协商） | 细节可与用户讨论调整 |
| **V** | Valuable（有价值） | 对用户或业务产生价值 |
| **E** | Estimable（可估算） | 团队能评估工作量 |
| **S** | Small（小） | 1-2个迭代可完成 |
| **T** | Testable（可测试） | 有明确的验收标准 |

## 验收标准

每个用户故事必须有明确的验收标准：

```markdown
**验收标准:**
1. [场景1：成功路径]
   - Given: 用户已登录
   - When: 用户点击"上传"按钮
   - Then: 弹出文件选择对话框

2. [场景2：文件类型限制]
   - Given: 用户选择文件
   - When: 文件格式为 .exe
   - Then: 显示错误提示"不支持该格式"

3. [场景3：文件大小限制]
   - Given: 用户选择超过10MB的文件
   - When: 点击确认上传
   - Then: 显示错误提示"文件大小超出限制"
```

## 故事拆分原则

当故事太大时，可以按以下方式拆分：

| 拆分方式 | 示例 |
|----------|------|
| 按操作步骤 | "上传文件" → "选择文件" + "确认上传" + "上传进度" |
| 按数据维度 | "管理所有用户" → "查看用户" + "编辑用户" + "删除用户" |
| 按用户类型 | "管理员功能" → "普通用户功能" + "管理员特有功能" |
| 按质量要求 | "功能完整" + "异常处理" + "性能优化" |

## 故事估算

常用估算方式：

| 方式 | 说明 |
|------|------|
| 斐波那契 | 1, 2, 3, 5, 8, 13, 21... |
| 衣服尺码 | XS, S, M, L, XL |
| T恤尺码 | XXS, XS, S, M, L, XL, XXL |

## 常见角色示例

| 角色 | 说明 |
|------|------|
| 访客 | 未注册/未登录的用户 |
| 注册用户 | 已完成注册的用户 |
| 付费用户 | 已购买服务的用户 |
| 管理员 | 后台管理人员 |
| 客服 | 客服人员 |

## 故事与需求的关系

```
需求文档（PRD）
    │
    ├── 需求点 1
    │       ├── 用户故事 1.1
    │       ├── 用户故事 1.2
    │       └── 用户故事 1.3
    │
    ├── 需求点 2
    │       ├── 用户故事 2.1
    │       └── 用户故事 2.2
    │
    └── 需求点 N
            └── ...
```

## 注意事项

1. **避免技术语言**：用户故事应由业务语言描述
2. **关注价值**：始终回答"为什么"而非"怎么做"
3. **保持简短**：单条故事不宜超过3行描述
4. **可测试**：必须有明确的完成定义（Definition of Done）
