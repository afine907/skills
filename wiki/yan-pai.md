# Yan-Pai - 迭代式代码审查Agent

【我要验牌】专业级迭代式代码审查与修复Agent，执行多轮"code review → report → bug fix"迭代流程。

## 核心特性

- 🔄 **迭代修复** - 自动发现问题、修复、验证的完整循环
- 📊 **多维审查** - 代码质量、潜在Bug、性能、安全、最佳实践
- 🎯 **智能判定** - "小瘪三"需擦皮鞋，"赌侠"质量达标
- 🔧 **自动修复** - 按优先级自动修复 medium 及以上问题

## 触发方式

```
验牌
我要验牌
审查代码
code review
检查一下
review一下
```

## 工作流程

```
开始 ──► Code Review ──► 生成报告 ──► 判定
                                      │
                          ┌───────────┴───────────┐
                          │                       │
                     有问题                    无问题
                          │                       │
                          ▼                       ▼
                     "小瘪三"                  "赌侠"
                          │                       │
                          ▼                       │
                       擦皮鞋                     │
                          │                       │
                          ▼                       │
                       Bug Fix                    │
                          │                       │
                          ▼                       │
                    迭代次数+1                     │
                          │                       │
                          ▼                       ▼
                    达到最大迭代? ────► 结束 ◄────┘
```

## 执行步骤

### Step 1: 初始化审查状态

创建审查状态追踪：
- 迭代次数: 0/{max_iterations}
- 当前状态: 待审查
- 历史报告: []
- 修复记录: []

默认 `max_iterations = 2`

### Step 2: 启动Code Review SubAgent

调用Agent工具进行全面代码审查：

| 维度 | 检查内容 |
|------|----------|
| 代码质量 | 可读性、命名规范、函数复杂度、代码重复 |
| 潜在Bug | 边界条件、空值处理、异常处理、并发安全 |
| 性能 | 算法复杂度、内存泄漏、不必要的计算 |
| 安全 | 输入验证、注入防护、敏感数据处理 |
| 最佳实践 | 设计模式、错误处理、测试覆盖 |

### Step 3: 解析审查报告

判定规则：

| 条件 | 判定 | 状态 |
|------|------|------|
| `issues.length > 0` | 小瘪三 | 需要擦皮鞋 |
| `issues.length === 0` 且 `passed: true` | 赌侠 | 任务完成 |
| 迭代次数 >= max_iterations | 赌侠 | 强制结束 |

### Step 4: 擦皮鞋 (Bug Fix)

按严重程度修复问题：
1. Critical - 必须立即修复
2. High - 尽快修复
3. Medium - 建议修复
4. Low - 可选修复

### Step 5: 迭代控制

- 未达最大迭代：继续审查
- 已达最大迭代：执行最终验收

## 严重性等级

| 等级 | 定义 | 示例 |
|------|------|------|
| Critical | 系统崩溃或严重安全漏洞 | SQL注入、空指针崩溃 |
| High | 影响核心功能或性能 | 内存泄漏、并发问题 |
| Medium | 影响代码质量或可维护性 | 代码重复、命名不规范 |
| Low | 轻微改进建议 | 注释优化、格式调整 |

## 评分标准

| 分数 | 状态 | 说明 |
|------|------|------|
| 90-100 | 赌侠 | 代码质量优秀 |
| 70-89 | 接近赌侠 | 有轻微问题 |
| 50-69 | 小瘪三 | 需要改进 |
| 0-49 | 严重小瘪三 | 需要重构 |

## 使用示例

```
# 启动验牌
验牌

# 指定迭代次数
验牌，迭代3次

# 审查特定文件
验牌 src/utils/helper.ts
```

## 注意事项

1. **迭代限制**: 默认2次，最多5次
2. **严重性阈值**: 只修复 medium 及以上问题
3. **人工介入**: 连续3轮无法修复的问题需人工处理
4. **上下文管理**: 每次不超过10个文件
